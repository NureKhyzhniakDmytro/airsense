# Сервисно-ориентированная архитектура системы управления вентиляцией

## Общая идея

Система управления вентиляцией описывается как сервисно-ориентированное решение с Kubernetes-развертыванием. Архитектура не дробит каждую сущность на отдельный микросервис, а выделяет только те компоненты, у которых есть самостоятельная ответственность, отдельный профиль нагрузки и смысл для независимого масштабирования.

Такой подход позволяет сохранить управляемую сложность проекта и одновременно показать инженерную зрелость системы: обработка телеметрии, автоматизация управления вентиляцией, пользовательское API и уведомления развиваются как отдельные компоненты.

## Состав системы

| Компонент | Ответственность | Причина выделения |
|---|---|---|
| API Service | REST API для веб- и мобильного клиентов, работа с пользователями, средами, комнатами, устройствами и настройками | Единая точка входа для клиентских приложений |
| Telemetry Ingestion Service | Прием MQTT-сообщений, валидация телеметрии, запись истории в PostgreSQL и обновление актуальных значений в Redis | Нагрузка зависит от количества IoT-устройств |
| Automation Service | Анализ телеметрии, применение правил и кривых регулирования, отправка команд вентиляционным устройствам | Алгоритмы управления масштабируются отдельно от API |
| Notification Service | Формирование и отправка push-уведомлений через Firebase Cloud Messaging | Изоляция внешней интеграции и повторных попыток отправки |
| Background Jobs Service | Агрегация истории, очистка старых данных, диагностические проверки | Плановые задачи не блокируют пользовательские запросы |

## Что остается внутри доменного ядра

Сущности `Environment`, `Room`, `Sensor`, `Device`, `UserRole` и связанные с ними правила доступа не выделяются в отдельные микросервисы на текущем этапе. Их разделение увеличило бы количество межсервисных вызовов, усложнило транзакции и не дало бы заметного выигрыша в масштабировании.

Поэтому они остаются внутри API Service или общего доменного ядра, а самостоятельными сервисами становятся только компоненты с действительно отдельным характером нагрузки.

## Kubernetes-развертывание

| Компонент | Kubernetes-ресурс | Назначение |
|---|---|---|
| API Service | Deployment, Service, HPA | Обработка клиентских запросов и горизонтальное масштабирование |
| Telemetry Ingestion Service | Deployment, Service, HPA | Масштабируемая обработка потока телеметрии |
| Automation Service | Deployment | Применение правил управления вентиляцией |
| Notification Service | Deployment | Отправка push-уведомлений и обработка ошибок FCM |
| PostgreSQL | StatefulSet или managed database | Основное хранилище данных |
| Redis | StatefulSet/Deployment или managed Redis | Кеш последних значений телеметрии |
| EMQX | StatefulSet или внешний MQTT-брокер | MQTT-коммуникация с IoT-устройствами |
| Ingress | Ingress Controller | Маршрутизация внешних HTTP/HTTPS-запросов |
| ConfigMap | ConfigMap | Нечувствительные параметры конфигурации |
| Secret | Secret | JWT, Firebase, PostgreSQL и MQTT credentials |

## Потоки данных

1. IoT-сенсор публикует телеметрию в MQTT-топик `sensors/{roomId}/telemetry`.
2. Telemetry Ingestion Service получает сообщение, валидирует его, сохраняет историческое значение в PostgreSQL и обновляет последнее значение в Redis.
3. Automation Service анализирует новое значение, применяет кривые регулирования и правила порогов.
4. Если требуется изменить режим вентиляции, Automation Service публикует команду в `devices/{deviceId}/control`.
5. Если значение критическое, Notification Service формирует push-уведомление и отправляет его через Firebase Cloud Messaging.
6. API Service предоставляет веб- и мобильному клиенту данные о комнатах, устройствах, телеметрии, настройках и истории.

## Роли пользователей

| Роль | Возможности |
|---|---|
| Владелец среды | Управляет участниками, комнатами, сенсорами, устройствами и правилами автоматизации |
| Администратор среды | Настраивает комнаты, параметры мониторинга и правила управления вентиляцией |
| Пользователь среды | Просматривает текущие показатели, историю телеметрии и получает уведомления |
| Неавторизованный пользователь | Может пройти регистрацию или вход в систему |

## Безопасность и наблюдаемость

Безопасность обеспечивается проверкой личности пользователя, разграничением доступа к средам и комнатам, защищенными HTTPS-каналами и контролируемым доступом к MQTT-топикам. Секретные параметры, включая ключи Firebase, строки подключения PostgreSQL и MQTT credentials, хранятся в Kubernetes Secret.

Для наблюдаемости используются журналирование, health checks, readiness/liveness probes и возможность подключения Prometheus/Grafana. Это позволяет контролировать состояние API, обработки телеметрии, автоматизации, MQTT-брокера и сервиса уведомлений.

## Stateful-компоненты

PostgreSQL, Redis и EMQX отличаются от stateless-сервисов тем, что зависят от состояния и данных. В демонстрационном контуре они могут быть описаны через StatefulSet, а в production-сценарии могут быть вынесены во внешние управляемые сервисы.

## Обоснование

Архитектура соответствует характеру предметной области. Пользовательские запросы, поток телеметрии, алгоритмы автоматизации и push-уведомления имеют разную интенсивность и разные требования к отказоустойчивости. Поэтому их логично разделить по сервисным границам.

Kubernetes используется как модель развертывания, которая обеспечивает:

- горизонтальное масштабирование API и обработки телеметрии;
- health checks для контроля состояния сервисов;
- rolling updates без остановки всей системы;
- централизованное управление конфигурацией через ConfigMap;
- безопасное хранение секретов через Secret;
- единый вход через Ingress.

## Вывод

Система представляется не как монолит, который планируется когда-то мигрировать, а как спроектированная сервисно-ориентированная платформа для управления вентиляцией. Такой подход делает дипломную работу сильнее: архитектура выглядит зрелой, но не искусственно переусложненной.
